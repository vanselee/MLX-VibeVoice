MLX Voice Notes 需求规格说明书 (PRD v2)
===

---

## 1. 项目概述

### 1.1 产品定义
一款基于 Apple Silicon 加速的 macOS 原生音频创作工具，面向短视频博主、播客主和内容创作者，提供从"文案/剧本"到"可发布语音音频"的极简工作流。

产品采用"本地开源为主，云端 API 为辅"的混合架构。Phase 0 负责验证本地 TTS、多人对话拼接和本地参考音频能力；Phase 0 通过后再制作低保真原型和细化 UI 流程。可发布 MVP 定义为 Phase 1 + Phase 2，核心目标是"多角色、多音色配音"，包含本地 TTS、角色解析、角色-音色绑定、分段生成、WAV 导出、可选字幕导出和任务队列。声音克隆不以"完美复刻"作为 MVP 承诺，但允许以"本地参考音色"形式进入 MVP；云端 API 先保留 Provider 抽象，不作为 MVP 依赖。

### 1.2 核心价值
- **隐私与成本**：优先利用本地 MLX 推理，降低联网依赖和 API 成本。
- **原生体验**：采用 SwiftUI 开发，符合 macOS 用户习惯。
- **生产力**：自动解析多人剧本，按角色音色批量生成音频。
- **可控质量**：默认优先生成速度，同时允许用户调节速度/质量优先级。
- **多音色配音优先**：产品不退化为单人旁白工具；即使早期 UI 简化，也必须围绕角色、音色和分段配音工作流设计。

### 1.3 目标用户
- 短视频创作者：重视快速生成、字幕、短时音频交付。
- 播客/有声内容创作者：重视多角色、长文本、连贯性和可恢复任务。
- 普通个人用户：希望本地、低成本地制作配音内容。

## 2. 功能详细说明

### 2.1 智能编辑器与文案解析

#### 2.1.1 编辑器形态
- MVP 使用简洁编辑器：`TextEditor` + 角色名行内高亮。
- 不做类 Notion block editor，避免早期编辑器复杂度失控。

#### 2.1.1.1 AI 文案整理提示词
- 编辑器需提供"AI 文案整理提示词"辅助入口，用于帮助用户在输入文案前，将普通文章整理为适合多角色配音的结构化脚本。
- MVP 不内置云端 AI 改写能力，不自动上传用户文案；仅提供可复制提示词模板，用户可粘贴到自己选择的 AI 工具中使用。
- 提示词模板需引导 AI 输出 `角色名：台词` 格式，并允许使用 `[停顿300ms]`、`[停顿500ms]`、`[慢速]`、`[快速]`、`[强调]`、`[升调]`、`[降调]` 等语音提示标记。
- 提示词需要求保留原文核心观点和事实，不编造新信息；输出只保留最终配音文案，不解释改写过程。
- 如果原文不适合多人配音，提示词应要求使用"旁白"作为默认角色。

默认提示词：

```text
请把下面这篇文章改写成适合多角色配音的软件输入文案。

要求：
1. 保留原文核心观点、事实和表达风格，不要编造新信息。
2. 将内容整理成适合朗读的短句，每段尽量不要太长。
3. 自动判断是否需要多个角色。如果适合多人配音，请加入角色名；如果不适合多人配音，则使用"旁白"作为默认角色。
4. 角色格式统一使用：角色名：台词
5. 如果出现未标记正文，请统一归入"旁白"。
6. 可以适当加入语音提示标记，但不要过度使用。可用标记包括：[停顿300ms]、[停顿500ms]、[慢速]、[快速]、[强调]、[升调]、[降调]。
7. 每个角色的说话风格要稳定。例如：
   旁白：清晰、客观、负责推进信息
   主讲人：自然、有观点、有情绪
   质疑者：提出反问或用户可能的疑虑
   案例角色：只在需要举例时出现
8. 输出只保留最终配音文案，不要解释你的改写过程。

原文如下：
【在这里粘贴你的文章】
```

#### 2.1.2 自动角色识别
- 支持解析 `角色名：内容`、`角色名: 内容` 或 `[角色名]内容`。
- 角色名识别前需做规范化：trim 首尾空白、统一全角/半角冒号、移除角色名内部多余空格。
- 软件自动提取候选角色列表，并尝试与音色库中的同名角色匹配。
- 首次识别到新角色时进入角色确认流程；用户可确认其为角色、绑定音色，或标记为普通文本/旁白。
- 对相似角色名提供合并提示，例如"小明"和"小 明"。
- 未匹配角色使用默认音色，并在角色面板中提示用户绑定音色。
- 检测到未带角色标记的正文时，弹出提示："有部分内容未标记音色，是否现在选择音色？"
- 用户选择现在处理时，可为未标记内容选择音色或绑定到已有角色。
- 用户选择忽略时，未标记内容默认归入"旁白"角色，使用旁白默认音色。

#### 2.1.3 实时预览
- 支持单句/单段播放。
- 点击段落旁试听按钮时，只生成当前段落音频。
- 同一段文本重复生成时，保留最近 3 个生成版本，方便回退。

#### 2.1.4 局部调整
- 允许选中特定文本重新生成。
- 重新生成默认不影响其他已完成段落。
- 支持用户手动选择某个历史版本作为当前版本。

#### 2.1.5 笔记组织方式
- MVP 使用平铺列表，按最近修改时间排序。
- 支持搜索标题和正文。
- 不做文件夹和标签系统。

#### 2.1.6 保存策略
- 自动保存采用防抖策略：停止输入 800ms 后保存。
- 切换笔记、关闭窗口、开始生成任务前强制保存。

#### 2.1.7 编辑器基础能力
- 支持系统级 Undo/Redo，快捷键为 `Cmd+Z` / `Shift+Cmd+Z`。
- 支持 `Cmd+S` 手动触发保存。
- 支持外部文本粘贴，统一规范化换行符、不可见控制字符和全角/半角冒号。
- MVP 编辑器需在 10000 字中文文本下保持可编辑、可滚动、无明显输入卡顿。

### 2.2 推理引擎

#### 2.2.1 本地 MLX 引擎 (Primary)
- Swift 原生优先集成 `mlx-audio-swift`。
- 使用模型能力矩阵决定可用功能，而不是假设所有模型都支持克隆、中文、多音色或流式输出。
- 生成默认优先速度；用户可切换速度/质量优先级。

#### 2.2.2 本地 Python/CLI 后备桥接
- 如果 Swift 版暂时无法稳定支持本地声音克隆，允许封装本地 Python/CLI 版 `mlx-audio` 作为后备。
- Python/CLI 桥接必须对用户透明：App 内统一下载/配置/调用，不要求用户手动进入终端。
- 桥接方案仅用于本地推理，不上传用户文本或参考音频。
- Phase 0 必须验证桥接可行性、依赖体积、启动耗时、错误处理和打包方式。
- 主 App 保持轻量，Python/CLI 运行组件优先按需下载：仅在用户首次使用本地克隆或相关模型时下载。
- 设置页展示 Python/CLI 组件体积、版本、安装位置和删除按钮。
- 如果 Swift 原生和 Python/CLI 两条本地克隆路线都不可行，Phase 0 结束时记录失败原因，并将 Phase 3 降级为"本地克隆实验功能延期"。
- 本地克隆延期后，不自动切换云端克隆；Phase 4 单独评估云端克隆替代方案、成本、隐私和服务商条款，再决定是否接入。

#### 2.2.3 云端 API 引擎 (Reserved)
- MVP 不接入云端服务商，不在主流程暴露云端入口。
- 从第一版架构中保留 TTS Provider 抽象，便于 Phase 4 接入 OpenAI、Azure、ElevenLabs、MiniMax、Fish Audio 等服务商。
- API Key 未来应存储在 Keychain，不得存入 UserDefaults 或明文配置文件。
- 云端成本控制需支持：单次任务预算、月度预算、预计费用、实际费用记录、超额阻断、本地缓存去重。

#### 2.2.4 失败处理策略
- 生成失败时只提示失败和错误原因，由用户手动重试。
- 错误类型至少区分：模型未下载、模型文件损坏、内存不足、生成超时、Python/CLI 桥接失败、音频导出失败、权限不足、磁盘空间不足、网络下载失败。
- 模型不支持某功能时按场景降级：
  - 不支持克隆：隐藏克隆入口或提示更换支持克隆的模型。
  - 克隆不可用：允许降级为预设音色。
  - 多音色不可用：提示用户拆分角色或更换模型。

#### 2.2.5 动态分段策略
- 不仅按 RAM 分段。分段需综合考虑模型上下文、句子边界、标点、预计生成时长、内存压力和历史失败记录。
- 默认优先速度，尽量减少过长上下文。
- 用户可切换速度/质量优先级；质量优先时允许更长上下文和更少拼接点。
- RAM 主要用于决定并发上限和模型加载策略，而不是唯一决定切片长度。
- 历史失败记录至少包含：模型名、分段字符数、预计音频时长、上下文长度、失败类型、内存压力摘要。
- 分段失败后，下一次同模型生成应降低单段字符上限；内存不足/OOM 类失败优先降低并发数。

### 2.3 角色与音色管理 (Speaker Hub)

#### 2.3.1 声音克隆
- 产品目标：支持本地参考音色/声音克隆；MVP 表述为"参考音色"或"基于参考音频生成相近音色"，不承诺 100% 复刻。
- Swift 原生 Qwen3-TTS 参考音频链路已在 Phase 0 跑通，真实参考音频主观听感通过，可作为 MVP 候选能力。
- 可用性以固定测试句的主观听感评估为准；不以量化相似度百分比作为发布承诺。
- 支持导入 5-30s 参考音频或 App 内实时录音。
- 参考音频格式：mp3、wav、m4a。
- 合规提示：不弹窗打断用户，仅在设置页提供免责声明和授权说明。

#### 2.3.2 音色库
- 管理系统预设音色和用户克隆音色。
- 支持为每个角色绑定默认音色。
- 角色-音色绑定支持跨笔记持久化：例如"小明"可在全局默认绑定到某个音色。
- 单篇笔记允许覆盖全局角色绑定；笔记内覆盖不影响其他笔记。
- 内置"旁白"角色，作为用户忽略未标记内容提示后的默认音色。
- 支持设置角色级语速、音量、音调。

#### 2.3.3 语音控制
- 支持全局语速/质量偏好。
- 支持角色级语速、音量、音调。
- 角色参数默认范围：语速 0.5x-2.0x（默认 1.0x），音量 -12dB 到 +6dB（默认 0dB），音调 -12 到 +12 半音（默认 0）。
- 支持文本标记：停顿、慢速、快速、强调、升调、降调、音调数值调整等。
- 标记示例：`[停顿500ms]`、`[慢速]`、`[快速]`、`[强调]`、`[升调]`、`[降调]`、`[音调+2]`。
- 多角色对话默认根据标点自动推断停顿，用户标记优先级高于自动推断。

#### 2.3.4 BGM
- MVP 不做 BGM 背景音轨。
- BGM 写入 roadmap，未来再评估混音、ducking 和分轨导出。

### 2.4 模型仓库 (Model Manager)

#### 2.4.1 可视化管理
- 提供内置推荐模型列表，包含描述、体积、语言、功能和适用硬件建议。
- 仅允许推荐模型列表，避免用户手动输入 Hugging Face 模型 ID 导致不可控失败。
- MVP 主路径是 App 内下载和管理推荐模型，不依赖用户本机已有模型。
- 手动导入/引用已有模型目录可作为后续高级功能保留，但不作为 MVP 下载失败的主要解决方案。

#### 2.4.2 下载系统
- 支持进度条、实时下载速度、断点续传、删除、更新管理。
- 支持模型文件完整性校验。
- 模型下载失败或不可用时，展示错误原因，并提供重新下载/删除损坏模型按钮。
- 下载系统必须优先解决国内外网络不稳定问题：支持可恢复下载、失败自动重试、手动重试、临时文件隔离和损坏缓存清理。
- 下载完成前不得把半截 `.safetensors` 视为可用模型；必须用文件大小、etag/hash 或 manifest 校验确认完整性。
- 推荐模型列表需记录每个模型的总下载大小、关键文件清单、校验信息和最低可用版本。
- 模型下载可考虑镜像源/备用端点配置，但必须在 UI 中明确来源和风险。

#### 2.4.3 硬件与模型推荐
- App 根据设备内存给出模型推荐范围，但不强制禁止用户选择超范围模型。
- 8GB 设备：推荐 0.6B 级模型；默认串行生成；参考音色和长文本生成需提示高内存风险。
- 16GB 设备：推荐 0.6B-1.7B 级模型；默认仍串行，可允许更长分段。
- 32GB+ 设备：可推荐更大模型或更高质量模式；并行生成仍需用户主动开启。
- 超出推荐范围时，模型卡片显示"不推荐当前设备使用"，说明可能出现生成慢、内存不足或系统卡顿。

#### 2.4.4 模型存储路径
- 用户可见目录：`~/Library/Application Support/MLXVoiceNotes/`。
- 设置界面展示具体目录路径。
- MVP 放弃 Mac App Store 沙盒路线。

#### 2.4.5 模型能力矩阵
Phase 0 必须建立模型能力矩阵，至少包含：

| 字段 | 说明 |
|-----|-----|
| 模型名 | Hugging Face repo 或本地模型标识 |
| Swift 原生支持 | 是否可通过 `mlx-audio-swift` 直接调用 |
| Python/CLI 支持 | 是否可通过本地 `mlx-audio` 调用 |
| 支持语言 | 中文、英文、日文等 |
| 声音克隆 | 是否支持 reference audio / zero-shot |
| 多音色 | 是否支持 voice selection |
| 流式生成 | 是否支持边生成边播放 |
| 推荐场景 | 短视频、播客、有声书、通用、克隆实验等 |
| 输出采样率 | 例如 16kHz、24kHz、44.1kHz |
| 模型大小 | 下载和磁盘占用 |
| 推荐硬件 | 8GB、16GB、32GB+ 的建议范围 |
| 内存占用 | 实测峰值；至少记录 M2 8GB，如有条件补充 16GB/32GB |
| 1000 字耗时 | M2 8GB 实测 |
| 中文效果 | 主观评分和问题记录 |
| 下载策略 | 是否支持断点续传、关键文件大小/hash、是否有镜像 |
| 许可证 | 是否允许商用、是否有模型限制 |

### 2.5 导出与联动

#### 2.5.1 音频导出
- MVP 只做 WAV 导出。
- MP3 放后续版本，不作为 MVP 验收项。
- 用户导出只提供整篇合成后的完整成品音频，不提供分段音频导出；分段音频仅作为内部生成、失败重试、版本缓存和拼接素材存在。
- WAV 导出必须支持常见播放器、剪辑软件和 Finder 预览。
- WAV 默认规格：16-bit PCM、mono、24kHz。
- 多角色或多模型拼接前，所有分段必须统一重采样到项目采样率；MVP 项目采样率固定为 24kHz。
- 重采样优先使用 AVFoundation/AudioConverter 等系统能力，要求避免明显爆音、变速或音高漂移。
- 响度标准：短视频模式 -14 LUFS，播客模式 -16 LUFS。
- 音频后处理：响度归一化、首尾静音裁剪、淡入淡出、多角色音量平衡。
- LUFS 测量与归一化优先采用 Apple 原生音频能力实现；如原生能力不足，Phase 0 评估是否引入轻量本地响度分析实现，避免为 MVP 引入完整 ffmpeg 依赖。
- 淡入淡出默认时长为 20ms，用于段落边界防止爆音；首尾淡入淡出默认 80ms。
- 低采样率模型输出可重采样到 24kHz，但不得宣称提升音质；模型能力矩阵需记录原始输出采样率和主观噪声问题。
- 导出文件名默认格式：`笔记标题_YYYYMMDD_HHMMSS.wav`。
- 导出位置通过系统保存面板选择；首次默认 Downloads 文件夹，之后默认使用用户上次选择的位置。

#### 2.5.2 字幕导出
- `.srt` 可选。
- MVP 采用句子级字幕，按标点和生成分段产生时间轴。
- SRT 文件使用 UTF-8 编码，文件名与音频同名，扩展名为 `.srt`。
- 后续可通过 forced alignment 升级到词级/字级时间戳。

#### 2.5.3 项目导出/导入
- 支持导出/导入单个笔记项目包。
- 支持导出/导入整个资料库备份。
- 单个项目包默认包含：文案、角色配置、生成参数和完整成品音频；不包含逐段音频文件。
- 克隆音色和原始参考音频支持随项目包导出，但必须由用户手动勾选。
- 导出克隆音色或原始参考音频时，必须在导出界面明确提示隐私和授权风险。

#### 2.5.4 开放接口
- CLI / REST API 不进 MVP，专注 GUI。
- CLI / REST 仅作为 Phase 5 可选项。

#### 2.5.5 持久化
- 所有设置具备记忆性，软件重启后自动载入。
- 普通开关可存 UserDefaults。
- API Key、未来云端凭证等敏感信息必须存 Keychain。

### 2.6 批量任务管理
- 任务队列采用两层结构：主队列以"篇/文案"为单位，任务详情以"段落"为单位。
- 主队列显示每篇文案的整体状态、总进度、失败数量、预计剩余时间和暂停/继续/取消/重试入口。
- 选中单篇文案后，详情区展示该文案的段落队列；每段显示角色、音色、版本、成功/失败状态和单段重试/跳过入口。
- 用户可设置并发数，但最大值受模型大小、设备内存和当前内存压力限制。
- 默认并发为 1。
- 任务界面显示总进度、当前处理段落、每段成功/失败状态、每段可单独重试/跳过、预计剩余时间和资源占用。
- 支持崩溃恢复和续跑；已完成段落不重复生成。

### 2.6.1 角色确认流程
- 角色确认使用非阻塞的内联面板或侧边栏列表，不使用逐个阻塞弹窗。
- 一篇文档首次解析出多个新角色时，批量展示候选角色，用户可逐个确认、合并、绑定音色或标记为普通文本/旁白。
- 用户拒绝某个候选角色后，对应文本默认回到未标记状态，并触发未标记内容处理流程；用户可选择音色或忽略为旁白。

### 2.7 缓存与资产管理
- App 数据根目录固定为 `~/Library/Application Support/MLXVoiceNotes/`。
- 建议目录结构：

```text
~/Library/Application Support/MLXVoiceNotes/
├── Models/
├── Projects/
├── Voices/
├── Cache/
│   ├── GeneratedSegments/
│   └── Downloads/
└── Exports/
```

- 每段文本最多保留最近 3 个生成版本。
- 生成缓存设置全局容量上限，默认 20GB；超过上限时自动清理最旧的未锁定缓存。
- 用户可在设置页调整缓存上限，并手动清理生成缓存、下载缓存和未使用模型。
- 设置页展示模型、生成缓存、音色、项目资产的磁盘占用。
- 删除笔记时必须询问用户是否同时删除关联生成音频和项目资产。
- 被用户手动选为当前版本或已导出的音频视为受保护资产，不应被自动缓存清理删除。
- 生成音频资产需记录状态：`isCurrentVersion`、`isExported`、`isStale`、`exportedAt`、`lastExportURL`。
- 用户通过 App 成功导出后，将对应音频标记为 `isExported = true`；即使用户之后在 Finder 中移动或删除导出文件，App 仍按已导出资产处理。
- 用户修改段落文本、角色音色或关键生成参数后，旧生成版本标记为 `isStale = true`；stale 版本可试听和手动保留，但不得作为默认当前版本参与整篇导出。

### 2.8 macOS 权限与本地资源访问
- 麦克风权限：仅在用户点击"录制参考音频"时请求。
- 文件导入权限：通过系统文件选择器导入 mp3、wav、m4a。
- 文件导出权限：通过系统保存面板导出 WAV/SRT。
- 本地写入：模型、缓存、项目资产写入 `~/Library/Application Support/MLXVoiceNotes/`。
- 网络访问：MVP 仅用于模型下载和更新推荐模型列表；未来云端 API 开启后再用于云端 TTS 请求。
- 非 Mac App Store 分发：使用 Developer ID 签名，并通过 Apple Notarization。
- 如果使用 Python/CLI 后备桥接，相关可执行文件、动态库和脚本必须由 App 管理，并纳入签名、公证和完整性校验流程。

### 2.9 本地日志与诊断
- App 记录本地日志，不自动上传。
- 日志存放在 `~/Library/Application Support/MLXVoiceNotes/Logs/`。
- 日志级别至少包含 info、warning、error。
- 生成失败时记录模型名、任务 ID、错误类型、分段长度、内存压力摘要和是否使用 Python/CLI；不得记录完整用户文案和参考音频内容。
- 设置页提供"导出诊断包"功能，包含日志、模型清单、系统信息、错误摘要、最近任务状态、段落数量、角色数量、分段数量、生成参数摘要。
- 诊断包不包含用户原始音频和完整文案；如需包含文本片段，仅允许记录截断后的前后各 20 个字符，并默认关闭。
- 日志文件需要轮转，单个日志文件默认不超过 10MB，最多保留 5 个历史日志文件。

### 2.10 应用生命周期与状态恢复
- App 重新打开后应恢复上次选中的笔记、窗口尺寸、窗口位置、侧栏折叠状态和右侧检查器状态。
- 编辑器滚动位置尽量恢复；如果恢复失败，不影响笔记内容加载。
- 生成任务状态需持久化，App 崩溃、强制退出或断电后，下次启动可识别未完成任务并提示继续、跳过或重新开始。
- 自动保存、任务状态和生成记录必须在关键步骤落盘，降低强制退出导致的数据丢失风险。

## 3. 用户界面 (UI) 设计规范

### 3.1 风格
- 纯原生 SwiftUI 视觉。
- 支持深色模式、SF Symbols 和 macOS 原生控件习惯。
- 不做营销式首页，启动后直接进入笔记/生成工作区。

### 3.2 布局
- **左侧栏**：笔记列表、模型仓库、音色库管理入口。
- **主视窗**：笔记编辑器、角色识别、高亮、段落试听和生成状态。
- **右侧/检查器面板**：角色音色、语速、音量、音调、生成参数。
- **底栏**：硬件资源占用、当前模型、分段策略、任务进度。
- 窗口最小尺寸建议为 1100x720。
- 左侧栏、右侧检查器面板可折叠；折叠状态需持久化。
- 支持不同分辨率和多显示器使用；窗口恢复时如上次显示器不可用，应回退到当前主屏幕可见区域。
- 低于推荐宽度时优先折叠右侧检查器和左侧栏，保证主编辑器可用。

### 3.3 原型策略
- 不在 Phase 0 之前制作完整高保真原型。
- Phase 0 技术验证通过后，先制作低保真原型和核心流程图，再进入 Phase 1 GUI 开发。
- 低保真原型需覆盖：笔记编辑、角色确认、未标记内容提示、音色绑定、任务队列、模型管理、导出保存面板、设置页缓存清理。

### 3.4 语言支持
- MVP 支持简体中文和英文界面。
- 首次启动默认跟随系统语言：系统首选语言为中文时使用中文；非中文系统默认使用英文。
- 设置页提供界面语言选项：跟随系统、中文、English。
- 用户手动选择语言后需持久化保存，后续启动优先使用用户选择。
- 模型名称、角色名、用户文案、文件路径和日志中的原始错误信息不强制翻译，避免误改用户内容或诊断信息。

## 4. 技术架构建议

### 4.1 开发环境
- Xcode 15+ / Swift 5.9+ / macOS 14+。
- Apple Silicon 专属，M1 起。

### 4.2 核心模块
- App Shell：SwiftUI。
- 数据层：SwiftData + UserDefaults + Keychain。
- 本地 Swift TTS：`mlx-audio-swift`。
- 本地后备 TTS/克隆：封装 Python/CLI `mlx-audio`，仅在 Swift 原生能力不足时启用。
- 音频播放/导出：AVFoundation / AudioToolbox 优先。
- 模型下载：URLSession Background。

### 4.3 数据存储
- SwiftData 存储笔记、段落、角色、全局角色-音色绑定、笔记内角色覆盖、任务、生成记录、模型索引。
- UserDefaults 存储非敏感偏好设置。
- Keychain 存储未来云端 API Key。
- 模型、音频缓存和项目资产存放在 `~/Library/Application Support/MLXVoiceNotes/`。

#### 4.3.1 SwiftData Schema 迁移
- 从 Phase 1 开始维护显式数据模型版本。
- 新字段必须提供默认值或可空策略，避免旧数据无法打开。
- 优先支持轻量自动迁移；涉及字段拆分、合并、关系重构时必须编写迁移计划并提供迁移测试。
- 新版本打开旧数据时应自动迁移；MVP 不要求旧版本 App 打开新版本数据。
- 迁移前如检测到高风险结构变更，应提示用户备份资料库。
- Phase 发布前必须使用上一阶段样例资料库执行迁移验证。

### 4.3.2 内存管理策略
- 模型按需加载，默认不常驻多个大模型。
- 同一时间优先只加载当前任务所需模型；切换模型时释放旧模型资源。
- M2 8GB 作为基线设备，默认推理并发为 1。
- Phase 0 必须实测不同模型在 M2 8GB 上的峰值内存，并写入模型能力矩阵。
- 内存压力升高时，优先暂停新任务启动、降低并发、缩短分段长度；不得静默降低音频质量。
- 发生内存不足/OOM 类失败后，任务应标记失败原因，并提示用户降低并发或更换轻量模型。

### 4.4 音频后处理
- LUFS 响度归一化；参考 ITU-R BS.1770-4，目标误差控制在 ±1 LUFS 内，Phase 0 需确定具体实现方式。
- 首尾静音裁剪；默认裁剪低于阈值的连续静音，阈值和最短静音时长在 Phase 0 实测后固定。
- 淡入淡出；段落边界默认 20ms，首尾默认 80ms。
- 多角色音量平衡；优先使用峰值和响度综合策略，避免单角色明显过大或过小。
- 段落拼接时根据自动停顿或文本标记插入静音。

### 4.5 节奏控制
- 自动推断：根据中文/英文标点、空行、角色切换决定停顿。
- 手动标记：支持 `[停顿500ms]`、`[慢速]`、`[快速]`、`[强调]`、`[升调]`、`[降调]`、`[音调+2]` 等轻量标记。
- 角色参数：语速、音量、音调。
- 模型不支持某个角色参数或文本标记时，应在 UI 中禁用该控制或提示该标记将被忽略。

### 4.6 云端 Provider 抽象
- 即使 MVP 不启用云端，也需要在架构上预留 `TTSProvider` 接口。
- Provider 至少抽象：输入文本、角色/音色、生成参数、成本估算、生成结果、错误码。
- 本地 Swift、本地 CLI、未来云端 API 都应尽量走统一任务模型。

### 4.7 Python/CLI 桥接打包约束
- Python/CLI 桥接只作为本地能力补足方案，优先用于声音克隆。
- 不要求用户手动安装 Python、Homebrew、ffmpeg 或命令行依赖。
- 依赖优先按需下载，而不是默认内置进主 App；下载过程必须有进度、失败重试和完整性校验。
- Phase 0 必须评估主 App 体积和 Python/CLI 组件体积；主 App 应尽量保持轻量，克隆组件作为可删除的本地组件管理。
- CLI 调用必须通过受控任务队列执行，禁止阻塞主线程。
- CLI 输入输出使用临时工作目录，任务完成后按缓存策略清理。
- CLI 错误必须转换为 App 内可读错误，不直接展示原始堆栈作为最终提示。
- 打包阶段必须验证签名、公证、Gatekeeper 启动、麦克风权限、文件导入和模型下载流程。

## 5. 开发路线图

### Phase 0：技术验证与能力矩阵
- 跑通 `mlx-audio-swift` 官方示例。
- 验证至少一个中文 TTS 模型可用。
- 验证多角色脚本拆分、分别生成、拼接 WAV 的完整链路。
- 验证真实参考音频生成链路，并由用户进行主观听感评估。
- 建立模型能力矩阵。
- 建立模型下载方案：断点续传、失败重试、完整性校验、损坏缓存清理和推荐模型 manifest。
- Python/CLI 桥接作为后备方向保留，不再阻塞 MVP 策略；仅在 Swift 原生能力不足时继续评估打包、依赖、启动耗时和错误处理。
- 实测 M2 8GB 上模型加载/卸载、峰值内存、默认并发上限和内存压力响应策略。
- 评估多模型输出采样率，并验证统一重采样到 24kHz 后的拼接质量。
- 确定重采样、LUFS 归一化、静音裁剪、淡入淡出的具体实现方案。
- 若 Swift 原生参考音频能力后续回归失败，记录失败原因，并将参考音色能力降级为实验功能。

**Phase 0 最低通过标准：**
- Swift 原生跑通至少一个中文 TTS 模型。
- 真实参考音频生成链路可用，且用户主观听感认为可接受。
- 中等长度文本可稳定生成并导出 WAV；长文本必须明确分段和内存策略。
- 多角色脚本可拆分后分别生成并拼接。
- 拼接后的 WAV 统一为 24kHz、16-bit PCM、mono。
- 模型下载方案具备明确实现路径，不依赖用户手动准备模型目录。

### Phase 1：GUI + 基础生成链路 + WAV
- 搭建 SwiftUI 主界面。
- 实现笔记列表、编辑器、自动保存。
- 接入本地 Swift TTS。
- 实现模型下载入口、下载状态、失败重试和完整性校验的基础版本。
- 支持至少一个默认音色/旁白生成，作为多角色生成的基础链路。
- 支持 WAV 导出。
- 支持基础 LUFS 归一化和静音裁剪。

### Phase 2：多角色多音色 + 字幕 + 任务队列
- 实现角色解析和角色音色绑定。
- 支持多角色、多音色配音，这是 MVP 的核心验收目标。
- 支持本地参考音色入口：导入参考音频、填写/校对参考文本、绑定到角色。
- 实现标点自动停顿。
- 支持句子级 `.srt` 字幕导出。
- 支持批量队列、任务状态、失败提示、手动重试。
- 显示预计剩余时间和资源占用。

### Phase 3：模型管理增强 + 克隆体验增强
- 完善本地参考音色/声音克隆体验，包括多条测试句、音色质量标记、参考音频版本管理。
- 评估是否需要 Python/CLI 后备桥接；如 Swift 原生已满足 MVP，则桥接可延后。
- 完善模型下载、断点续传、校验、删除、更新。
- 完善动态分段和并发限制。
- 可选加入手动导入/引用已有模型目录，服务高级用户和开发者，不作为普通用户主路径。

### Phase 4：云端 API 预留能力落地
- 接入第三方 API Provider。
- 实现成本估算、预算上限、缓存、实际消耗记录。
- 明示云端隐私风险和服务商数据政策。

### Phase 5（可选）：CLI / REST
- CLI 接口。
- 本地 REST API。

## 6. 非功能性需求

### 6.1 性能基线
- M2 8GB 上，1000 字中文文本生成 WAV 耗时目标 ≤ 60s。
- 默认速度优先模式下，短视频 3 分钟以内脚本应能稳定完成。
- 播客/长文本模式允许更慢，但必须支持续跑和任务恢复。

### 6.2 目标系统范围
- macOS 14+。
- Apple Silicon 专属，M1 起。
- 不支持 Intel Mac。

### 6.3 自动更新与崩溃上报
- MVP 不做自动更新。
- MVP 不做崩溃上报。
- 用户手动下载新版。
- 设置页可提供"检查新版本"按钮，打开或检测项目发布页；MVP 不做后台自动检查。

### 6.4 隐私与合规
- 本地模式下文本、参考音频、生成音频不上传。
- 云端模式未来上线时，必须在 UI 中明示服务商是否可能存储、训练或审核用户文本/音频。
- 设置页提供声音克隆免责声明。
- 导出包含克隆音色或参考音频的项目包时，必须提示隐私和授权风险。

### 6.5 稳定性
- 长任务中断后可续跑。
- 已完成段落不重复生成。
- 模型文件损坏可检测、删除和重新下载。
- 并发数必须受内存压力限制，避免 App 或系统崩溃。

### 6.6 产品方向与许可证
- 竞品分析、开源协议和商业模式不阻塞 Phase 0 技术验证。
- Phase 0 后、公开发布或上传 GitHub 前，必须确定开源协议和授权边界。
- 竞品分析至少覆盖 ChatTTS、CosyVoice、Fish Speech、ElevenLabs、PlayHT 和 macOS 系统 TTS。
- 商业模式可选项包括：完全个人工具、开源项目、捐赠、Pro 版、云端 API 增值；MVP 阶段暂不实现付费能力。

## 7. 验收标准

### 7.1 Phase 1 验收
- 可创建、编辑、保存笔记。
- 自动保存防抖为停止输入 800ms；`Cmd+S` 可手动保存，`Cmd+Z` / `Shift+Cmd+Z` 可撤销/重做。
- 10000 字中文文本下编辑器可输入、滚动和保存。
- 可使用至少一个本地 Swift TTS 模型生成中文语音。
- 推荐模型可在 App 内下载；下载中断后可继续或重试；半截模型不会被当作可用模型。
- 可导出 WAV。
- 导出的 WAV 为 24kHz、16-bit PCM、mono，可被 Finder、QuickTime 和常见剪辑软件识别。

### 7.2 Phase 2 验收
- 可解析多人对话格式。
- 新角色识别需经过确认流程；误识别的"时间"、"地点"等可标记为普通文本或旁白。
- 相似角色名会提示合并。
- 可为不同角色绑定不同音色。
- 检测到无角色标记文本时，会提示用户选择音色；用户忽略后归入"旁白"角色。
- 全局角色-音色绑定可跨笔记复用，单篇笔记可覆盖该绑定。
- 可为不同角色绑定不同预设音色或本地参考音色。
- 至少一个真实参考音频生成结果由用户主观判断为可用。
- 可生成多角色拼接音频。
- 可导出句子级 `.srt`。
- 批量任务界面可显示进度、失败状态、预计剩余时间和资源占用。

### 7.3 Phase 3 验收
- 本地参考音色/声音克隆体验增强：支持多条测试句、音色版本管理和质量标记。
- 模型下载支持进度、断点续传、完整性校验、损坏重下、删除和更新。
- 可选支持手动导入/引用已有模型目录。
- 长文本生成可续跑。

### 7.4 标准测试样例
- 单人中文短文：200 字，验证单音色生成、WAV 导出、响度归一化。
- 多角色对话：至少 3 个角色、10 轮对话，验证角色识别、音色绑定、停顿推断和拼接。
- 长文本：1000 字中文，验证动态分段、任务进度、续跑和性能基线。
- 标记文本：包含 `[停顿500ms]`、`[慢速]`、`[快速]`、`[强调]`、`[升调]`、`[降调]`、`[音调+2]`，验证文本标记解析。
- 参考音频：5s、15s、30s 三档，验证导入、录音和克隆稳定性。
- 失败场景：未下载模型、模型损坏、磁盘空间不足、麦克风权限拒绝、网络中断、Python/CLI 失败。
- 导出场景：首次导出默认 Downloads；再次导出默认上次选择位置；可同时导出 WAV 和可选 SRT。
- 角色歧义场景：包含 `时间：2026年4月29日`、`地点：北京`、`小明：你好`，验证角色确认和误识别纠正。
- 缓存一致性场景：生成后修改段落文本，旧音频标记为 stale，不参与默认整篇导出。
- 诊断场景：一次模拟生成失败后，可导出不含完整文案和原始音频的诊断包。
- 后处理场景：不同采样率分段可统一重采样到 24kHz，段落边界无明显爆音、变速或音高漂移。

### 7.5 测试策略
- 单元测试覆盖：角色解析、角色名规范化、分段策略、缓存状态转换、文件命名、成本估算 Provider 接口。
- 集成测试覆盖：SwiftData 保存/读取/迁移、模型索引、任务队列、生成记录、项目导入导出。
- 端到端测试覆盖：单音色生成、多角色拼接、SRT 导出、失败重试提示、诊断包导出。
- 性能回归测试覆盖：1000 字生成耗时、10000 字编辑器性能、M2 8GB 内存峰值、模型加载/卸载耗时。
- Python/CLI 桥接测试覆盖：组件下载、完整性校验、进程超时、错误码转换、临时目录清理。
- 每个 Phase 发布前必须跑通对应 Phase 的验收样例。

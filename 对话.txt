## 一、整体评价

**优点**
- 定位明确：本地优先 + 云端兜底的混合架构，符合"成本/隐私/性能"三角的最优解。
- 选型务实：选 `mlx-audio-swift` 而非自己接 PyTorch/MLX Python，避开了跨语言桥接的工程坑。
- MVP 分期合理：先跑通单音色 → 多角色 → 模型管理 → 云端，是正确的递进路径。
- 注意到了成本控制和缓存，这点很多同类需求文档会漏掉。

**主要问题**
1. **对 `mlx-audio-swift` 的能力假设过于乐观**——文档把它当成"什么都能做"的黑盒，但该库当前对**声音克隆 / Zero-shot voice cloning**的支持是有边界的（取决于底层模型，如 Kokoro、CSM、Spark-TTS、Dia 等，每个模型克隆能力差异很大）。需要先做一次"模型能力矩阵"调研，再写需求。
2. **多人对话不只是"正则解析 + 多音色"**——还涉及**对话节奏、情感、停顿、轮次过渡**，TTS 合成时如果只是逐句拼接，会非常机械。需要明确 MVP 是否接受"机械拼接"。
3. **"动态分段按 RAM 调整字数"是伪需求**——切片大小应由模型上下文窗口、显存峰值、韵律连贯性决定，而不是 RAM。8GB 和 32GB 的 Mac 跑同一个模型，单次推理的合理切片长度其实差不多，差别在并行度。
4. **缺少"音频后处理"环节**——响度归一化（LUFS）、静音裁剪、淡入淡出、多角色音量平衡都没提，但这是"能听"和"能用"的分水岭。
5. **REST API + CLI + GUI 三套接口在 MVP 阶段是负担**，建议 Phase 4 只保留一项。

---

## 二、查漏补缺（建议补入需求文档）

### A. 必须补充的功能性需求
| 缺失项 | 建议 |
|---|---|
| **SSML 或标记语言支持** | 至少支持 `<break time="500ms"/>`、语速/音调标记，否则多人对话节奏死板 |
| **对话停顿策略** | 角色切换时的默认间隔（如 300ms）、可配置 |
| **音频后处理** | LUFS 归一化（播客 -16 LUFS / 短视频 -14 LUFS）、首尾静音裁剪、淡入淡出 |
| **导出格式** | 仅 mp3 不够，建议加 wav（无损，便于二次剪辑）和分轨导出（每个角色一条音轨，便于在剪映/FCP 中调整） |
| **字幕/时间戳导出** | 同步生成 .srt 或 .json 时间轴，短视频博主刚需 |
| **失败重试与降级** | 本地推理失败 → 提示是否降级到云端；云端失败 → 重试策略 |
| **克隆音色的合规提示** | 上传参考音频时弹一次"确认拥有授权"的勾选，规避法律风险 |

### B. 必须补充的非功能性需求
| 缺失项 | 建议 |
|---|---|
| **性能基线** | 例如"M2 8GB 上，1000 字 Kokoro 推理 ≤ 60s"，否则验收无依据 |
| **模型存储路径** | 是否走 App Sandbox？模型动辄几 GB，需明确放 `~/Library/Application Support/...` 还是用户可选目录 |
| **隐私声明** | 云端模式下文案是否会被服务商训练？需在 UI 里明示 |
| **崩溃恢复** | 长任务中途崩溃，已生成的分段是否保留 |
| **并发控制** | 批量队列同时跑几个？避免内存炸掉 |

### C. 技术架构上的建议修正
1. **存储层**：Core Data 对笔记型应用偏重，建议直接用 **SwiftData**（macOS 14+ 已 GA，与你的目标系统匹配）或 SQLite + GRDB。Realm 已被 MongoDB 收购后维护节奏放缓，新项目不推荐。
2. **本地 HTTP Server**：Swifter 多年未更新，建议 **Hummingbird** 或 **Vapor**（如果只做本机回环可以用 Network.framework 自己写一个极简的，避免重依赖）。
3. **mlx-audio-swift 的实际形态**：建议在 Phase 1 之前先做一个 **2 天的 spike**——跑通官方 Demo，确认：(a) 哪些模型可用；(b) 克隆是否真的 Zero-shot；(c) 长文本切片后韵律是否连贯；(d) Apple Silicon 各代性能差异。这个 spike 的结论会直接重写 Phase 2/3。
4. **云端 API 选型**：OpenAI TTS 不支持声音克隆；要做克隆得用 **ElevenLabs**、**MiniMax**、**Fish Audio**、**Azure Custom Neural Voice**，建议在文档里明确各家"克隆 / 多语言 / 价格 / 延迟"的对比表，再决定接哪几家。
5. **成本控制**应再细化：月度预算上限、单次任务上限、超额阻断、按角色显示消耗，而不只是"发送前预估"。

### D. 需求里语义模糊的地方
- "**笔记型交互（类 Notion）**"——是要做 block-based 编辑器（很重）还是普通富文本？强烈建议 MVP 用 `TextEditor` + 行内角色高亮，不要碰 block editor。
- "**多版本结果对比**"——是 A/B 试听？还是历史版本回滚？UI 完全不同。
- "**预留 BGM 背景音轨空间**"——架构上"预留"通常意味着不做，建议直接砍掉这句话，避免误导后续开发。

**Q14 是关于"你写的文案/剧本"在 App 里怎么组织。**

把它想成 macOS 自带的"备忘录"或者 Bear、Obsidian——你会在 App 里写很多个文案（每个文案 = 一篇笔记），时间一长会有几十上百篇，需要某种方式找到它们。

四种方案的差异：

**A. 平铺列表（按时间排序）**
左侧栏就是一长串笔记标题，按"最近修改时间"倒序排。
- 优点：零操作成本，写完就走
- 缺点：笔记多了之后只能靠搜索找
- 适合：你预计自己同时在做的项目不多（< 30 篇）

**B. 文件夹/分组**
左侧栏可以建文件夹，比如「短视频脚本」「播客 EP01」「客户 A 的项目」，每篇笔记归到一个文件夹里。
- 优点：结构清晰，符合 Mac 用户习惯
- 缺点：需要手动归类，一篇笔记只能在一个文件夹

**C. 标签 tag**
不分文件夹，给每篇笔记打多个标签，比如同一篇笔记可以同时是 `#短视频` `#美妆` `#客户A`，左侧按标签筛选。
- 优点：灵活，一篇笔记可以从多个维度找到
- 缺点：需要养成打标签习惯，新用户上手慢

**D. 文件夹 + 标签都要**
两套并存，像 Bear、Things 那样。
- 优点：最灵活
- 缺点：UI 复杂度上一个台阶，开发量大约是 A 的 3 倍

---

我的建议：如果你主要是个人使用、文案数量可控，选 **A**；如果预期会按"项目/客户"长期管理，选 **B**。MVP 阶段不建议 C/D。
